# F1 Fantasy Expected Points — Project Specification

## What This Project Does

Computes expected fantasy points for each of the 22 F1 drivers by:
1. Fetching betting odds from The Odds API (race winner, podium, top 6, etc.)
2. Supplementing with manually-entered odds from Oddschecker when API markets are unavailable
3. Devigging the odds using Shin's method to get fair probabilities
4. Fitting a Plackett-Luce ranking model to recover full finishing position distributions
5. Running 50,000 Monte Carlo race simulations
6. Computing expected points under our family league scoring rules
7. Displaying results on a web dashboard with an interactive methodology explainer

## Scoring Rules (Family League)

This is NOT the official F1 Fantasy game. Our family league uses simplified scoring:

**Grand Prix race finish:**
- P1=25, P2=18, P3=15, P4=12, P5=10, P6=8, P7=6, P8=4, P9=2, P10=1, P11-P22=0
- DNF/NC = **-20 points**

**Sprint race finish (6 weekends: China, Miami, Canada, Great Britain, Netherlands, Singapore):**
- P1=8, P2=7, P3=6, P4=5, P5=4, P6=3, P7=2, P8=1, P9+=0
- DNF/NC = **-20 points**

**That's it.** No qualifying points, no positions gained, no overtakes, no fastest lap bonus, no Driver of the Day. Each player picks 5 drivers before qualifying each weekend (same drivers for sprint + race). No budget cap.

## Architecture

```
GitHub Actions (Python)          Vercel (Static Site)
┌────────────────────┐          ┌──────────────────┐
│ Cron: Fri 6am UTC  │          │ Vite + React app │
│ or manual trigger  │          │                  │
│                    │  commit  │ Reads:           │
│ pipeline/update.py ├─────────►│ /data/latest.json│
│ - fetch odds (API) │  push    │                  │
│ - devig (Shin)     │          │ Auto-redeploys   │
│ - fit PL model     │          │ on push          │
│ - simulate 50K     │          └──────────────────┘
│ - write JSON       │
└────────────────────┘
```

**Why this split:** The Plackett-Luce optimizer + 50K simulations takes 30-60 seconds and needs numpy/scipy. Vercel Hobby has a 10-second function timeout. So GitHub Actions does the heavy compute, commits the JSON result, and Vercel just serves the static site. Push triggers auto-redeploy. Total cost: $0/month.

**Why not Supabase:** No user-generated data, no auth, no real-time needs. The entire "database" is one JSON file regenerated ~24 times/year. Git gives you version history for free.

## Implementation Details

### Pipeline (Python)

All source files are provided in `pipeline/`. Here's what each does:

**`config.py`** — Constants: 22 drivers with team assignments, team colors, scoring tables, sprint weekends, driver name matching map.

**`devig.py`** — Three devig methods (Shin's preferred, multiplicative and power as alternatives). Shin's method accounts for favorite-longshot bias by solving for an insider trading parameter z. All methods take an array of implied probabilities and return fair probabilities summing to 1.

**`odds_fetcher.py`** — Two data sources:
- **The Odds API** (the-odds-api.com): Auto-discovers F1 sport key via `/v4/sports`, fetches race winner odds (`outrights` market), attempts additional markets (`outrights_top3`, `outrights_top6`, `outrights_top10`) via event-specific endpoint. Free tier = 500 req/month.
- **Manual JSON**: Files in `public/data/odds_input/` with American odds per driver per market. Manual overrides API for same market, supplements for missing markets.
- Returns: `observed_probs` dict mapping market name → {driver_index: fair_probability}

**`plackett_luce.py`** — The core model:
- **Plackett-Luce model**: Each driver has strength λ_i. P(driver i wins from remaining set S) = λ_i / Σ λ_j. Full finishing order drawn sequentially.
- **DNF as mixture**: With probability p_dnf_i, driver DNFs. Otherwise finishing position drawn from PL. DNF decomposes into team reliability + driver component.
- **Team structure**: log(λ_i) = μ_team[t(i)] + δ_driver[i]. Teammates share team strength, creating natural positive correlation.
- **Fitting**: scipy L-BFGS-B optimizer minimizes squared error between model-implied cumulative probabilities and observed odds. Each eval runs ~20K PL simulations. Regularization: teammate similarity + mild shrinkage toward mean.
- **Final output**: 50K simulations → position distribution array (23 values: P1..P22 + DNF), expected points (race + sprint), standard deviation.

**`update.py`** — Entry point. CLI args: `--manual FILE`, `--api-key KEY` (or `ODDS_API_KEY` env var), `--output DIR`. Orchestrates: fetch odds → devig → fit PL → simulate → write JSON.

### Frontend (React + Vite)

The frontend needs to be created. It should have two pages:

**Page 1: Expected Points Dashboard**
- Fetch `/data/latest.json` on load
- Table of all 22 drivers sorted by expected total points
- Each row shows: rank, driver name, team color indicator, mini sparkline of position distribution (P1-P22), expected points with bar gauge, P(Win), P(Podium), P(Top 10), P(DNF), standard deviation
- Sort options: by E[points], safest (highest P(top10)), upside (highest P(win)), lowest variance
- Toggle for sprint weekend (adds sprint expected points)
- Metadata footer: race name, date, model info, last updated

**Page 2: Methodology**
- Step-by-step walkthrough: (1) Convert odds → fair probs, (2) Fit Plackett-Luce, (3) Simulate 30K races, (4) Compute expected points
- Interactive driver selector → shows that driver's full position distribution (bar chart, hoverable), CDF curve with overlay dots at odds cutoff points (P1, P3, P6, P10), expected points decomposition by position
- Lambda (strength parameter) visualization for all drivers
- Correlation explanation: team effects + chaos/race volatility
- Scoring reference table

**Design direction:** Dark theme (#0a0a0a background), monospace data font (JetBrains Mono), display font (Syne or similar), team colors from config. F1-inspired but not cluttered. Dense data presentation — this is a tool for a family league, not a marketing site.

## The Odds API Details

- **Sign up**: https://the-odds-api.com (free tier: 500 requests/month)
- **F1 sport key**: Use the `/v4/sports` endpoint to discover — likely `motorsport_formula_one` or similar
- **Race winner market**: `outrights` — this is the primary and most reliable market
- **Additional markets**: Try `outrights_top3`, `outrights_top6`, `outrights_top10` via the event-specific endpoint `/v4/sports/{sport}/events/{eventId}/odds`
- **If a market isn't available from the API**, supplement with manual JSON from Oddschecker. The pipeline merges both sources (manual overrides API for same market).
- **Regions**: Use `us,uk,eu,au` for maximum bookmaker coverage

## Key Technical Notes

- **The PL optimizer uses Monte Carlo simulation inside the objective function.** This means each optimizer evaluation runs ~20K race simulations. It's stochastic, so we use different random seeds per eval to smooth the loss surface. Total fitting takes ~100-200 evaluations × 20K sims = 2-4M simulated races.
- **The final output uses 50K simulations** with a fixed seed for reproducibility.
- **Shin's devig method** solves for an insider trading parameter z via bisection. It's more accurate than simple multiplicative normalization for F1 odds because favorites are underpriced and longshots are overpriced (favorite-longshot bias).
- **The position_distribution array has 23 elements**: indices 0-21 are P(finish in position 1-22), index 22 is P(DNF).
- **Team correlation is structural**, not a separate parameter: teammates share log(λ) team component + regularization penalizes large teammate gaps. In simulation, if one teammate is strong, they occupy top positions, leaving fewer for competitors — but their teammate benefits from the same car.
- **The client-side simulation** (for the Methodology page interactive charts) is a lighter JS re-implementation that runs in the browser. It reads the fitted λ and p_dnf from the JSON and re-simulates ~10-30K races for the interactive distribution charts. This avoids needing a server for interactivity.

## What's NOT Implemented Yet

These are stretch goals, not blockers:
- **Head-to-head odds**: Oddschecker offers driver matchup odds (P(A beats B)). In PL, P(A beats B) = λ_A/(λ_A+λ_B) — very tight constraint on relative strength. Could add as additional objective terms.
- **Qualifying model**: Separate PL fit using pole/qualifying odds for more accurate positions-gained modeling. Not needed for our league (no qualifying points).
- **Chaos/rain model**: Race-level volatility mixture (normal σ vs high σ). Currently all races use the same noise level.
- **Season projector**: Accumulate per-race expected points across the season to project WDC standings.
- **Lineup optimizer**: Given expected points, find the optimal 5-driver combination.
